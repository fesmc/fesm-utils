# Adopting `coords` in climber-x / yelmo (unified `map_class`, cdo-first)

Plan for moving the downstream models (climber-x, yelmo, yelmox) onto the new
`coords` library in `fesm-utils/utils`. Downstream migrates to the **unified
`map_class` + `map_field`** API for all remapping; **`cdo` stays as the
conservative weight *generator*** behind it, until the analytic clip is proven
fast enough in production. The `coords` redesign is documented in
[coords-design.md](coords-design.md); remaining library work in
[coords-followups.md](coords-followups.md).

## Goal & constraint

- Downstream uses the **unified `map_class`** everywhere: build a map once, apply
  with `map_field` / `nc_read_interp(map=…)`. The legacy SCRIP applier
  (`map_scrip_field`) and the `mps=` form of `nc_read_interp` are **not** carried
  forward into model code — the call sites are migrated.
- **Conservative weights keep coming from `cdo gen…`.** `cdo` is hidden behind a
  conservative `map_init`; the result is loaded into the unified `weight_map`
  store like any other map.
- This is safe because **the generation method and the apply path are
  decoupled**: cdo and the analytic clip both produce the same `weight_map`, and
  `map_field` applies it identically. "cdo now, analytic later" is a one-flag
  change at generation time, invisible to call sites.

## Downstream usage inventory

Counted across the model trees (call sites):

| Symbol | climber-x | yelmo | yelmox | climber-x-orig/yelmo1 |
|---|---|---|---|---|
| `nc_read_interp*` | — | **1294** | 63 | 208 |
| `map_scrip_field*` | **122** | 18 | 6 | 30 |
| `map_scrip_init*` | 40 | 30 | 15 | 14 |
| `map_field` / `map_init` | 3 | — | — | 1 |

Takeaways:
- Downstream mapping is almost entirely SCRIP + `cdo` today; the IDW/`map_field`
  path is barely used. The migration is therefore mostly a **mechanical call-site
  conversion** from the SCRIP surface to the unified one.
- yelmo's dominant call is `nc_read_interp` (read a NetCDF field and remap onto
  the model grid). It is *not* "load+apply only": via `map_scrip_init` /
  `nc_read_interp` it can trigger `cdo` generation. The `cdo` call lives inside
  the mapping library, not model source.

## What `coords` already provides (verified)

- **Unified read+interp parity.** [ncio_interp.f90](../src/coords/ncio_interp.f90)
  exposes `nc_read_interp(map=…)` for `dp/sp/int` 2D & 3D and `logical` 2D — the
  full set the downstream trees call. `map_field` (grid→grid) carries the
  fill/smoothing/`mask_pack`/`reset` options the SCRIP applier had
  (see coords-followups.md §2).
- **cdo → `map_class` load bridge.** [mapping.f90](../src/coords/mapping.f90)
  `map_read(map, src_name, dst_name, fldr, method)` loads a SCRIP file (incl. a
  `cdo gen…` one) into a `map_class` via `map_scrip_to_weight_map`.
- **cdo generation.** [mapping_scrip.f90](../src/coords/mapping_scrip.f90)
  `map_scrip_init` shells out `cdo gen<method>` (generate-or-load) and writes the
  SCRIP file.

## The one coords-side seam to add

There is no **single** call that generates a conservative map via `cdo` and
returns a `map_class`; today you compose `map_scrip_init` (generate the file) +
`map_read` (load it). Add that convenience so downstream has one entry point and
the cdo→analytic flip is a single switch:

```fortran
! conservative map into the unified store, generated by cdo (cached to file)
call map_init(map, grid1, grid2, method="con", gen="cdo")   ! gen="clip" later
```

Concretely: a conservative `map_init` (or `map_init_conservative(..., gen=…)`)
that, for `gen="cdo"`, calls `map_scrip_init` then folds the result into
`map%wm` — reusing the existing `map_read` / `map_scrip_to_weight_map` bridge.
`gen="clip"` routes to the analytic
[conservative.f90](../src/coords/conservative.f90) path. Default stays `cdo`.

This is the **only** new library surface the migration needs; everything else is
call-site conversion downstream.

## Phased plan

**Phase 1 — land the cdo→`map_class` seam (in `coords`).**
- Add the conservative `map_init(..., method="con", gen="cdo")` entry point above.
- Confirm `nc_read_interp(map=…)` + `map_field` cover every dtype/rank/option the
  downstream trees use (inventory above; `logical` only appears as 2D).
- *Gate:* `coords` test suite stays green; a cdo-generated `map_class` reproduces
  the same weights as the direct SCRIP load.

**Phase 2 — migrate downstream call sites to the unified API (per tree).**
- Replace `map_scrip_init` + `map_scrip_field` / `nc_read_interp(mps=…)` with the
  conservative `map_init` (cdo) + `map_field` / `nc_read_interp(map=…)`.
- Apply the one-time struct-field renames the redesign introduces
  (`grid%mtype` → `grid%cs%mtype`, etc. — see coords-design.md).
- Generation is still `cdo`, so results should match the pre-migration run.
- *Gate per tree:* builds, links, reproduces a reference run within tolerance.

**Phase 3 — flip generation to the analytic clip (later, gated).**
- Once benchmarks ([coords-performance.md](coords-performance.md)) show the clip
  is production-fast, switch the conservative `map_init` default from `gen="cdo"`
  to `gen="clip"`. Call sites and apply are unchanged.
- **Off until explicitly enabled** — the "keep cdo for now" requirement.

## Cleanup after migration

Once all trees are on the unified API (end of Phase 2), the legacy SCRIP applier
surface in `coords` — `map_scrip_field` (already labelled "testing reference
only") and the `mps=` `nc_read_interp` family — has no production consumer and
can be removed, leaving `mapping_scrip` as the SCRIP **reader/writer + cdo
generator** only.

## Validation

- Phase 1: `coords` suite green; cdo-`map_class` vs direct-SCRIP-load weight diff = 0.
- Phase 2: per-field tolerance diff against the pre-migration (SCRIP) run, per tree.
- Phase 3: area-conservation invariant + `cdo gencon` diff (already in the suite)
  before enabling `gen="clip"` in any production config.

## Open items

- Settle the conservative `map_init` signature (`gen=` string vs a logical
  `use_cdo=`; how the cdo cache filename is derived — reuse `gen_map_filename`).
- Confirm no downstream tree needs a `logical` 3D `nc_read_interp` (none seen).
- Sequence the trees: yelmo first (largest `nc_read_interp` surface) or climber-x
  first (drives the shared build) — decide when starting Phase 2.
